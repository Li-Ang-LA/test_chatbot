# Git Project Management Rules

These rules apply to ALL files. Enforce the team's git conventions automatically.

## Identity (HIGHEST PRIORITY)

- Never include "Generated by AI", "Sent via Claude", or any AI tool signatures in any output
- No AI-related metadata or disclaimers in Issues, PRs, commits, code comments, or docs
- Write as a human developer: concise, professional, no "As an AI..." phrasing

## When creating a branch

- Format: `<type>/<issue-number>-<short-description>`
- Type must be one of: feat, fix, docs, style, refactor, test, chore, ci, perf, revert
- Description: lowercase, words separated by `-`, 2‚Äď4 words max
- Always branch from latest `main`: run `git switch main && git pull` first
- Examples: `feat/12-user-auth`, `fix/34-login-crash`, `refactor/56-db-layer`

## When writing commit messages

- Format: `<type>(<scope>): <imperative subject>`
- Type: same list as branch types
- Scope: the module or area touched (e.g. `api`, `ui`, `auth`, `db`, `config`, `ci`, `deps`)
- Subject: English, imperative mood ("add" not "added"), lowercase start, no period, ‚Č§72 chars
- Footer: use `Refs #N` on intermediate commits; use `Closes #N` only on the final commit before PR
- Body (optional): explain WHY, not WHAT

## When to commit

- One commit = one atomic change that can be described in a single sentence
- Each commit must pass lint/format checks and not introduce syntax/import errors
- **DO commit when:** a single logical unit is complete ‚ÄĒ a new function, a config addition, a test, a focused bugfix
- **Do NOT commit when:** code is half-finished, multiple unrelated changes are mixed together, or debug residue (`print`, `console.log`, `pdb`, `breakpoint()`) remains
- **Do NOT bundle** unrelated changes: "fix login bug + rename variables + update README" = 3 separate commits
- Prefer slightly finer granularity on feature branches ‚ÄĒ squash merge compresses them into one clean commit on `main`
- If a commit is hard to describe in one `<type>(<scope>): <subject>` line, it's too large ‚ÄĒ split it

## When creating Issues

- Title format: `[Type] description` where Type is capitalized
- Examples: `[Feat] Add SSO login`, `[Bug] Crash on empty form submit`, `[Refactor] Split monolith auth module`
- **Required labels:** `type:` √ó 1 + `priority:` √ó 1
- **Recommended labels:** `scope:` √ó 1~2 when specific modules are involved
- `status:` labels are dynamic ‚ÄĒ add/remove as state changes (e.g. `status:blocked`, `status:needs-repro`)

## When creating Pull Requests

- Title format: `<type>(<scope>)[#<issue-number>]: <subject>`
- If no corresponding Issue, omit `[#N]`: `<type>(<scope>): <subject>`
- Examples: `feat(auth)[#12]: add SSO login flow`, `chore(ci): add lint to CI pipeline`
- Body MUST include `Closes #N` to auto-close the linked Issue
- **Required labels:** `type:` √ó 1 (match the Issue/branch type)
- **Recommended labels:** `scope:` √ó 1~2 (match the Issue)
- **Do NOT add** `priority:` labels to PRs ‚ÄĒ these belong at the Issue level
- Add reviewers and milestone
- Use Draft PR for work-in-progress

## When merging

- Always use squash merge: `gh pr merge --squash --delete-branch`
- Never merge without at least 1 approval and passing CI
- After merge, clean up local branch: `git switch main && git pull && git branch -d <branch>`

## Code quality gates before push

- Run the project's lint/format checks (whatever the repo uses)
- Run the project's test suite
- No hardcoded secrets, credentials, or environment-specific paths
- No debug prints, breakpoints, or commented-out code blocks
- No large binary files (build artifacts, datasets, media) unless explicitly tracked
